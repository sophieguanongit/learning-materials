01什么是 Docker：集装箱革命

🐳 一句话理解 Docker

Docker 就像"软件集装箱"——把应用程序和它需要的所有依赖打包在一起，确保在任何环境都能一致运行。

从运输集装箱说起

在 Docker 诞生之前，软件部署就像"搬家"：

🏠 传统的搬家方式

家具、电器、衣物混装，到了新家还要重新整理。每家的布局不同，东西可能放不下。

📦 集装箱式搬家

所有物品按房间打包进标准箱子。不管运到哪里，打开箱子就能恢复原样。

Docker 的 Logo 是一只鲸鱼驮着集装箱，这完美诠释了它的理念：标准化打包，随处运行。

"给我一个装满代码的集装箱，我能把它运到任何服务器上运行。"

— Docker 社区格言

Docker vs 虚拟机

对比维度 Docker 容器 虚拟机
启动速度 秒级 分钟级
资源占用 MB 级 GB 级
隔离级别 进程级隔离 操作系统级隔离
性能损耗 接近原生 有一定损耗
适用场景 微服务、CI/CD、开发环境 强隔离需求、多操作系统

🤔 产品经理需要记住的

Docker 让"在我电脑上能跑"成为历史。它把应用和环境的依赖打包在一起，保证了从开发到测试到生产的环境一致性。

02为什么需要 Docker

它解决什么问题

❌ 没有 Docker 的痛点

  • "在我电脑上是好的，怎么上线就挂了？"
  • 新同事配置开发环境要一整天
  • 测试环境和生产环境配置不一致
  • 同时维护多个项目，依赖版本冲突
  • 部署流程复杂，容易出错

✅ 有 Docker 之后

  • 开发、测试、生产环境完全一致
  • 一条命令启动完整开发环境
  • 每个项目独立容器，互不干扰
  • 部署就是"复制粘贴"容器
  • 版本回滚秒级完成

真实的场景

场景一：新人入职

传统方式：装 Node、Python、MySQL、Redis... 一天过去了。Docker 方式：docker-compose up，5 分钟开始写代码。

场景二：项目切换

项目 A 用 Node 16，项目 B 用 Node 18。传统方式：nvm 切来切去。Docker 方式：每个项目独立容器，互不影响。

场景三：线上故障

本地复现不了？直接把线上的 Docker 镜像拉到本地运行，环境完全一致，问题一目了然。

场景四：快速扩容

流量高峰来了？启动 10 个相同的容器，负载均衡自动分发。流量下去就关掉，按需付费。

💡 核心价值

一致性（开发=测试=生产）+ 效率（秒级启动/部署）+ 隔离性（项目互不干扰）= Docker 的核心竞争力

03核心概念速查

四大核心概念

概念 类比 说明
Image（镜像） 📦 集装箱的设计图纸 只读模板，包含运行应用所需的一切：代码、运行时、库、配置
Container（容器） 🏠 盖好的房子 镜像的运行实例，相互隔离，轻量级（可以同时跑很多个）
Volume（数据卷） 💾 外接硬盘 持久化存储，容器删除后数据不丢失
Network（网络） 🔗 局域网 容器间通信的桥梁，同一网络的容器可以互相访问

镜像 vs 容器：一张图理解

# 镜像是静态的"模具"
Image: nginx:latest  # 一个 Nginx 镜像（约 140MB）

# 容器是镜像的"活体"
Container:
  - web-server-1  # 从 nginx 镜像启动的容器 #1
  - web-server-2  # 从 nginx 镜像启动的容器 #2
  - web-server-3  # 从 nginx 镜像启动的容器 #3

一个镜像可以创建无数个容器，就像一张光盘可以复制无数份。

常用镜像仓库

Docker Hub

官方公共仓库，有 nginx、redis、mysql、node 等官方镜像，免费使用。

私有仓库

企业内部镜像存储，如阿里云容器镜像服务、Harbor。

GitHub Container Registry

与 GitHub 集成，CI/CD 自动构建推送镜像。

云厂商镜像服务

AWS ECR、Google GCR、Azure ACR，与云原生服务深度集成。

Dockerfile：镜像的"配方"

Dockerfile 是一个文本文件，描述如何构建镜像：

# 从官方 Node 镜像开始
FROM node:18-alpine

# 设置工作目录
WORKDIR /app

# 复制依赖文件
COPY package*.json ./

# 安装依赖
RUN npm install

# 复制源代码
COPY . .

# 暴露端口
EXPOSE 3000

# 启动命令
CMD ["npm", "start"]

04Docker Compose：多容器编排

🚀 什么是 Docker Compose

Docker Compose 是一个工具，让你用一个 YAML 文件定义和运行多个容器。不用一个个手动启动，一条命令搞定整个应用栈。

典型应用架构

一个典型的 Web 应用可能包含：

🌐 Web 服务

Nginx 反向代理，处理静态资源和负载均衡

⚙️ 应用服务

Node.js / Python / Java 后端 API

🗄️ 数据库

PostgreSQL / MySQL 存储业务数据

⚡ 缓存

Redis 缓存热点数据，提升响应速度

docker-compose.yml 示例

version: '3.8'

services:
  # 前端 Nginx
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf
    depends_on:
      - api

  # 后端 API
  api:
    build: ./api
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgres://db:5432/myapp
    depends_on:
      - db
      - redis

  # 数据库
  db:
    image: postgres:15
    volumes:
      - postgres_data:/var/lib/postgresql/data
    environment:
      - POSTGRES_DB=myapp
      - POSTGRES_PASSWORD=secret

  # 缓存
  redis:
    image: redis:alpine

volumes:
  postgres_data:

常用命令

命令 作用
docker-compose up -d 后台启动所有服务
docker-compose down 停止并删除所有容器
docker-compose ps 查看运行状态
docker-compose logs -f 查看日志（实时）
docker-compose restart 重启所有服务

05常见使用场景

1. 本地开发环境

场景

团队统一开发环境，避免"我这边能跑"的问题。

# 克隆项目后，一条命令启动
docker-compose up -d

# 所有依赖就绪：
# - MySQL 在 localhost:3306
# - Redis 在 localhost:6379
# - 后端 API 在 localhost:3000

2. CI/CD 流水线

场景

自动化测试和部署，每次代码提交自动构建镜像、运行测试、推送仓库。

# GitHub Actions 示例
- name: Build Docker Image
  run: docker build -t myapp:${{ github.sha }} .

- name: Run Tests
  run: docker run myapp:${{ github.sha }} npm test

3. 微服务架构

场景

每个微服务一个容器，独立开发、部署、扩容。

用户服务

Container A，端口 3001

订单服务

Container B，端口 3002

支付服务

Container C，端口 3003

通知服务

Container D，端口 3004

4. AI/ML 环境

场景

打包 CUDA、PyTorch、模型权重，一键部署推理服务。

# 带GPU支持的推理容器
FROM nvidia/cuda:11.8-pytorch

RUN pip install transformers fastapi uvicorn
COPY ./model /app/model
CMD ["uvicorn", "main:app", "--host", "0.0.0.0"]

与开发沟通的语言

产品经理应该能说清楚

  • 环境问题："这个 bug 在本地 Docker 能复现吗？"
  • 部署需求："这个功能需要新加什么服务/容器？"
  • 性能问题："容器资源够吗？需要扩容吗？"
  • 数据问题："这个数据存在 Volume 里还是容器内部？"
  • 时间估算："部署上线大概要多久？包含哪些步骤？"

常见术语

术语 含义
镜像（Image） 应用的打包模板，包含代码和依赖
容器（Container） 镜像的运行实例
编排（Orchestration） 管理多个容器的生命周期（如 Kubernetes）
镜像仓库（Registry） 存储和分发镜像的地方（如 Docker Hub）
Dockerfile 定义如何构建镜像的脚本
docker-compose.yml 定义多容器应用的配置文件

07实战案例解读

案例：一个 AI Agent 服务的 Docker 配置

这是一个实际的 OpenClaw AI Agent 项目的简化版配置：

docker-compose.yml

version: '3.8'

services:
  # 主 Agent 服务
  openclaw:
    image: node:18-alpine
    container_name: openclaw-main
    working_dir: /app
    volumes:
      - ./src:/app/src          # 代码热更新
      - ./config:/app/config    # 配置文件
      - ./logs:/app/logs        # 日志持久化
    environment:
      - NODE_ENV=production
      - ANTHROPIC_API_KEY=${ANTHROPIC_API_KEY}
    ports:
      - "3000:3000"
    command: npm start
    restart: unless-stopped

  # Redis 缓存（对话历史）
  redis:
    image: redis:alpine
    container_name: openclaw-redis
    volumes:
      - redis_data:/data
    ports:
      - "6379:6379"

volumes:
  redis_data:

配置解读

volumes（挂载）

把本地目录映射到容器内，代码改了不用重新构建镜像。

environment（环境变量）

敏感信息（API Key）通过环境变量传入，不写死在代码里。

ports（端口映射）

左边是宿主机端口，右边是容器端口。访问 localhost:3000 就能访问服务。

restart（重启策略）

unless-stopped 表示除非手动停止，否则崩溃后自动重启。
