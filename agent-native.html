01什么是 Agent Native

核心定义

Agent Native 指的是为 AI Agent（智能体）原生设计的数据格式、工具接口和工作流程。就像 "Cloud Native" 是为云环境设计的应用，Agent Native 是让 AI Agent 能够最高效理解、处理和生成的产品设计范式。

为什么格式选择如此重要

在传统软件开发中，我们为人类用户设计界面和文档。PDF、Word、PPT 这些格式优化的是人类阅读体验——固定排版、丰富视觉、打印友好。

但 AI Agent 的工作方式完全不同：

Agent 不需要视觉排版

Agent 通过 API 和结构化数据交互，固定的页面布局反而增加解析成本

Agent 需要程序化处理

能够被代码解析、验证、转换，而不是依赖 OCR 和模糊匹配

Agent 需要确定性输出

Schema 验证确保输出格式稳定，下游系统可以可靠处理

Agent 需要链式调用

一个 Agent 的输出成为另一个 Agent 的输入，需要标准化格式

格式层级：从机器友好到人类友好

排名 格式 Agent 友好度 核心优势
1 JSON / 结构化数据 最佳 Schema 验证、Function Calling 原生支持、下游自动化
2 Markdown 优秀 LLM 原生输出、版本控制友好、轻量解析
3 HTML 中等 语义结构清晰、支持交互，但噪音较多
4 PDF / Word 较差 固定排版为人类设计，机器解析困难
"The future of software is agents talking to agents. And agents speak JSON."

— 行业共识

02JSON：机器优先的结构化语言

为什么 JSON 是 Agent 的首选

JSON (JavaScript Object Notation) 已成为 Agent Native 系统的基础数据交换格式。它的优势在于：

✓ 优势

  • Schema 验证：可以定义严格的 JSON Schema，确保数据结构正确
  • Function Calling 原生支持：OpenAI、Anthropic 的工具调用都基于 JSON
  • 类型安全：支持字符串、数字、布尔值、数组、对象等类型
  • 广泛工具链：所有编程语言都有成熟的 JSON 库
  • 嵌套结构：可以表达复杂的层级关系

✗ 局限

  • 人类可读性差：复杂嵌套时难以阅读
  • 无注释支持：标准 JSON 不支持注释
  • 文件体积：相比二进制格式较大
  • 长文本不友好：大量文本内容不适合用 JSON 存储

JSON Schema：Agent 输出的契约

JSON Schema 是 Agent Native 系统的核心组件，它定义了 Agent 输出必须遵循的结构：

// OpenAI Structured Outputs 示例
{
  "name": "extract_user_info",
  "parameters": {
    "type": "object",
    "properties": {
      "name": { "type": "string" },
      "age": { "type": "integer", "minimum": 0 },
      "email": { "type": "string", "format": "email" },
      "interests": {
        "type": "array",
        "items": { "type": "string" }
      }
    },
    "required": ["name", "email"]
  }
}

Function Calling：Agent 与工具的桥梁

OpenAI 的 Function Calling 和 Anthropic 的 Tool Use 都基于 JSON Schema，让 Agent 能够：

  1. 声明能力：定义 Agent 可以调用的函数及其参数
  2. 验证输入：确保调用参数符合预期格式
  3. 结构化输出：函数返回值也是结构化 JSON
  4. 链式调用：一个函数的输出可以直接作为另一个函数的输入
🛠️ 产品设计启示

当你的产品需要与 AI Agent 集成时，优先提供 JSON API 而非人类友好的 HTML 页面。Agent 不需要漂亮的 UI，它需要的是稳定、可预测的数据结构。

MCP (Model Context Protocol)：Agent 通信的新标准

Anthropic 主导的 MCP 协议正在成为 Agent 与外部工具交互的标准。它基于 JSON-RPC，定义了：

  • Resources：Agent 可以读取的数据源
  • Tools：Agent 可以调用的功能
  • Prompts：预定义的提示模板

MCP 的意义在于：任何实现了 MCP 的工具都可以被任何支持 MCP 的 Agent 使用，打破了厂商锁定。

03Markdown：LLM 的原生输出

为什么 Markdown 是 Agent 的第二语言

Markdown 的设计哲学是 "易读易写"，这恰好与 LLM 的输出特性完美匹配：

LLM 原生输出格式

所有主流 LLM（GPT、Claude、Gemini）都默认输出 Markdown，无需额外转换

轻量级解析

解析器简单，几乎所有语言都有实现，处理速度快

版本控制友好

纯文本格式，Git diff 清晰可读，适合 AI 生成内容的迭代

人类与机器兼顾

既是 Agent 可靠解析的结构化文本，也保持良好的可读性

CLAUDE.md：Anthropic 的最佳实践

Anthropic 在 Claude Code 产品中引入了 CLAUDE.md 文件格式，这是 Agent Native 文档设计的典范：

# Project Name

> Project description in one line

## Build Commands

- Build: `npm run build`
- Test: `npm test`
- Lint: `npm run lint`

## Code Style

- Use TypeScript strict mode
- Prefer functional components
- Follow existing patterns in the codebase

## Architecture

Brief description of key components...

CLAUDE.md 设计原则

  • 简洁优先：避免冗余，Claude 能从代码中推断的不需要写
  • 结构清晰：使用标题和列表组织信息
  • 命令可执行：提供具体的构建、测试命令
  • 约定优于配置：遵循项目已有模式

Markdown 在 Agent 工作流中的角色

场景 Markdown 的优势
文档生成 LLM 直接输出格式化文档，无需后处理
代码注释 README、API 文档天然支持 Markdown
知识库 Obsidian、Notion 等工具原生支持，易于 Agent 读写
报告输出 结构化标题 + 列表 + 代码块，适合生成分析报告
PR 描述 GitHub PR 模板使用 Markdown，Agent 可自动生成
"Markdown is the lingua franca of AI output. It's structured enough for machines to parse, readable enough for humans to understand."

— AI Engineering 社区

04格式对比与选择指南

场景决策矩阵

使用场景 推荐格式 原因
Agent 工具调用 / API 返回 JSON Schema 验证、类型安全、下游自动化
LLM 输出文档 / 报告 Markdown 原生输出、可读性好、易存储
Agent 间数据交换 JSON 标准化、工具链完善
知识库 / 文档存储 Markdown 版本控制、检索友好
Web 内容展示 HTML 语义化、交互能力
最终交付给人类阅读 PDF 排版固定、打印友好

格式的组合使用

在实际系统中，多种格式往往组合使用：

📦 典型工作流示例
  1. Agent 处理：读取 JSON 配置 → 执行任务 → 输出 JSON 结果
  2. 结果转换：JSON → Markdown 报告（供人类审阅）
  3. 最终交付：Markdown → HTML/PDF（供终端用户阅读）

关键洞察：格式转换应该是单向的——从机器友好流向人类友好，而不是反过来。

避免的陷阱

✗ 常见错误

  • 让 Agent 输出 PDF/Word（格式控制困难）
  • 用 JSON 存储长篇内容（可读性差）
  • 用 Markdown 做结构化数据交换（缺乏验证）
  • 将 PDF 解析为 Agent 输入（OCR 不可靠）

✓ 最佳实践

  • Agent 间通信：始终用 JSON
  • 人类审阅：用 Markdown 中间层
  • 最终交付：按需转换为 PDF/HTML
  • 知识存储：Markdown + Git 版本控制

05行业趋势与专家观点

Anthropic：Tool Use 与 Agent 设计

"When building with Claude, think in terms of structured outputs. Use tool definitions with clear JSON schemas. This makes your agents more reliable and easier to debug."

— Anthropic Engineering Blog

Anthropic 的核心观点：

  • 结构化优先：设计 Agent 系统时，先定义 JSON Schema
  • CLAUDE.md 模式：用 Markdown 文件作为 Agent 的上下文配置
  • MCP 协议：推动 Agent 与工具的标准化通信

a16z：Agent Native 产品形态

"The most successful AI-native products won't just add AI to existing workflows—they'll reimagine the entire product around what AI agents can do best."

— a16z: What Is an AI Agent?

a16z 的核心洞察：

  • 重新设计产品：不是给现有产品加 AI，而是围绕 Agent 能力重新设计
  • 数据格式即基础设施：Agent Native 需要新的数据格式基础设施
  • 从 Chatbot 到 Agent：从对话式交互进化到自主执行任务

swyx：AI Engineering 的兴起

"AI Engineering is not just about prompting—it's about building systems where LLMs are components. Data formats and structured outputs are the glue that holds these systems together."

— swyx, AI Engineering 概念提出者

swyx 强调的趋势：

  • LLM 作为组件：不是魔法，而是需要可靠集成的基础设施
  • 结构化输出是关键：让 LLM 输出可被下游系统处理
  • 从 Demo 到 Production：可靠性比炫酷更重要

Google：结构化数据与 Agent

Google 在 Agent 领域的布局：

  • Gemini Function Calling：基于 JSON Schema 的工具调用
  • Vertex AI Agent Builder：企业级 Agent 开发平台
  • 结构化输出 API：强制 LLM 输出符合 Schema

开源社区：数据基础设施演进

Apache Iceberg

为数据湖提供表结构，让 Agent 可以可靠地查询和修改大规模数据

Delta Sharing

开放协议，让不同系统的 Agent 可以安全共享结构化数据

LangChain / LlamaIndex

Agent 框架，内部大量使用 JSON 进行状态管理和工具调用

MinIO

对象存储，配合 Agent 处理非结构化数据向结构化数据转换

06产品经理实践指南

产品设计检查清单

当设计 Agent 集成功能时，问自己：

  • ✅ 我的 API 返回的是 JSON 还是 HTML？
  • ✅ 我是否提供了清晰的 JSON Schema 文档？
  • ✅ Agent 调用的错误信息是否结构化？
  • ✅ 长文本内容是否用 Markdown 而非 JSON 存储？
  • ✅ 我是否考虑了 Agent 间的数据流转？
  • ✅ 最终交付给人类的格式是什么？需要转换吗？

技术决策框架

决策点 选择依据
API 响应格式 JSON（始终），考虑 GraphQL 增加灵活性
Agent 配置文件 Markdown（CLAUDE.md 模式）或 YAML
日志与追踪 JSON Lines（便于日志分析工具处理）
知识库存储 Markdown + Git，或向量数据库 + 元数据 JSON
报告生成 Markdown 中间格式 → 按需转 HTML/PDF
跨系统数据交换 JSON + JSON Schema 验证

与开发团队沟通的语言

产品经理应该能说清楚：

  1. 数据流向：从哪个 Agent 输出，到哪个 Agent 输入
  2. 格式约束：每个环节使用什么格式，为什么
  3. 验证规则：数据的哪些字段是必须的，类型是什么
  4. 错误处理：格式不符时 Agent 应该如何响应
  5. 人类介入点：在哪个环节需要人类审阅/确认

避免的 Product Anti-patterns

✗ Anti-patterns

  • "让 AI 直接输出 PDF"
  • "用自然语言描述 API 参数"
  • "Agent 输出格式看情况"
  • "用户要什么格式就输出什么"

✓ 正确做法

  • JSON → Markdown → PDF 分层转换
  • JSON Schema 定义清晰参数
  • 输出格式作为产品契约的一部分
  • 内部格式标准化，外部按需适配
