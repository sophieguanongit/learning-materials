01为什么需要 JSON Schema

核心定义

JSON Schema 是一份契约。它定义了 JSON 数据必须遵循的结构、类型和约束规则。在 Agent Native 系统中，JSON Schema 是 AI 与软件系统之间最可靠的沟通协议。

问题：没有 Schema 的世界

假设你让 AI 提取用户信息，输出如下：

{
  "name": "张三",
  "age": "25",
  "email": "zhangsan@example.com"
}

看起来没问题？但仔细看：age 是字符串 "25" 而不是数字 25。下游系统期望数字，收到字符串——程序崩溃。

更糟糕的情况：

// AI 有时会"发挥创意"
{
  "姓名": "张三",           // 字段名用了中文
  "年龄": "二十五岁",       // 值也用了中文
  "邮箱地址": "暂无",        // 缺少有效邮箱
  "备注": "这是一个VIP用户"  // 多了未预期的字段
}

解决：Schema 的力量

定义一个 Schema，告诉 AI 输出必须遵循的规则：

{
  "type": "object",
  "properties": {
    "name": { "type": "string" },
    "age": { "type": "integer", "minimum": 0 },
    "email": { "type": "string", "format": "email" }
  },
  "required": ["name", "email"],
  "additionalProperties": false
}

有了这个 Schema，AI 的输出就被约束了：

类型正确

age 必须是整数，不能是字符串

字段固定

只能输出 name、age、email，不能自创字段

必填校验

name 和 email 必须存在

格式验证

email 必须符合邮箱格式

JSON Schema 的价值

场景 没有 Schema 有 Schema
API 开发 接口文档靠人工维护，容易过时 Schema 即文档，代码即校验
AI 输出 LLM 输出不稳定，后处理复杂 强制输出符合预期结构
数据交换 格式不一致，调试困难 双方约定清晰，错误快速定位
Agent 通信 Agent 间无法可靠传递数据 标准化协议，链式调用稳定
"JSON Schema is the contract between AI and software. It turns fuzzy LLM outputs into reliable API inputs."

— AI Engineering 实践者

02基础语法速成

Schema 的基本结构

每个 JSON Schema 至少需要一个 $schema 声明和 type 定义：

{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "string"
}

这个最简 Schema 表示：数据必须是一个字符串。

七种基本类型

类型 说明 示例
string 字符串 "Hello"
number 数字（整数或小数） 42, 3.14
integer 整数 42
boolean 布尔值 true, false
null 空值 null
array 数组 [1, 2, 3]
object 对象 {"key": "value"}

字符串约束

{
  "type": "string",
  "minLength": 1,
  "maxLength": 100,
  "pattern": "^[a-zA-Z]+$",
  "format": "email"
}
📝 常用 format

email · uri · date · date-time · uuid · hostname · ipv4 · ipv6 · phone

数字约束

{
  "type": "integer",
  "minimum": 0,
  "maximum": 100,
  "exclusiveMinimum": 0,
  "multipleOf": 5
}

数组约束

{
  "type": "array",
  "items": { "type": "string" },
  "minItems": 1,
  "maxItems": 10,
  "uniqueItems": true
}

对象约束

{
  "type": "object",
  "properties": {
    "name": { "type": "string" },
    "age": { "type": "integer" }
  },
  "required": ["name"],
  "additionalProperties": false
}

⚠️ 关键设置：additionalProperties

设置 "additionalProperties": false 可以禁止额外字段。在 AI 输出场景中非常重要，防止 LLM "发挥创意"添加未预期的字段。

组合类型

// 允许多种类型
{ "type": ["string", "null"] }

// 枚举值
{
  "type": "string",
  "enum": ["draft", "published", "archived"]
}

// 常量
{ "const": "v1.0" }

03高级特性

嵌套对象

Schema 可以无限嵌套，表达复杂的数据结构：

{
  "type": "object",
  "properties": {
    "user": {
      "type": "object",
      "properties": {
        "name": { "type": "string" },
        "address": {
          "type": "object",
          "properties": {
            "city": { "type": "string" },
            "country": { "type": "string" }
          }
        }
      }
    }
  }
}

$ref：引用与复用

使用 $ref 引用其他 Schema，避免重复：

{
  "type": "object",
  "definitions": {
    "address": {
      "type": "object",
      "properties": {
        "city": { "type": "string" },
        "country": { "type": "string" }
      }
    }
  },
  "properties": {
    "billingAddress": { "$ref": "#/definitions/address" },
    "shippingAddress": { "$ref": "#/definitions/address" }
  }
}

条件逻辑：if/then/else

{
  "type": "object",
  "properties": {
    "type": { "enum": ["personal", "business"] }
  },
  "if": { "properties": { "type": { "const": "business" } } },
  "then": {
    "required": ["companyName", "taxId"]
  },
  "else": {
    "required": ["firstName", "lastName"]
  }
}

组合关键字

关键字 含义 示例场景
allOf 必须满足所有条件 合并多个 Schema
anyOf 满足任一条件即可 多种可选的数据结构
oneOf 必须恰好满足一个条件 互斥的数据类型
not 不能满足该条件 排除特定模式 
// anyOf 示例：允许字符串或数字
{
  "anyOf": [
    { "type": "string", "format": "date" },
    { "type": "string", "format": "date-time" }
  ]
}

复杂示例：商品 Schema

📦 完整的商品数据 Schema
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "properties": {
    "id": { "type": "string", "format": "uuid" },
    "name": { "type": "string", "minLength": 1, "maxLength": 200 },
    "price": { "type": "number", "exclusiveMinimum": 0 },
    "currency": { "type": "string", "enum": ["CNY", "USD", "EUR"] },
    "tags": {
      "type": "array",
      "items": { "type": "string" },
      "uniqueItems": true
    },
    "status": { "enum": ["draft", "active", "inactive"] },
    "variants": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "sku": { "type": "string" },
          "color": { "type": "string" },
          "size": { "type": "string" },
          "stock": { "type": "integer", "minimum": 0 }
        },
        "required": ["sku"]
      }
    },
    "metadata": {
      "type": "object",
      "additionalProperties": true
    }
  },
  "required": ["id", "name", "price", "currency"],
  "additionalProperties": false
}

04在 AI Agent 中的应用

OpenAI Structured Outputs

OpenAI 的 Structured Outputs 功能强制模型输出符合 JSON Schema：

// OpenAI API 调用示例
{
  "model": "gpt-4o",
  "messages": [...],
  "response_format": {
    "type": "json_schema",
    "json_schema": {
      "name": "user_profile",
      "strict": true,
      "schema": {
        "type": "object",
        "properties": {
          "name": { "type": "string" },
          "interests": {
            "type": "array",
            "items": { "type": "string" }
          }
        },
        "required": ["name"],
        "additionalProperties": false
      }
    }
  }
}

🔑 strict: true 的意义

设置 "strict": true 后，OpenAI 保证输出100%符合 Schema。这是从"尽量符合"到"必须符合"的质变。

Anthropic Tool Use

Anthropic 的 Tool Use 同样基于 JSON Schema：

// Claude Tool Use 定义
{
  "name": "search_products",
  "description": "搜索商品数据库",
  "input_schema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "搜索关键词"
      },
      "category": {
        "type": "string",
        "enum": ["electronics", "clothing", "books"]
      },
      "max_price": {
        "type": "number",
        "description": "最高价格"
      }
    },
    "required": ["query"]
  }
}

Function Calling 的设计模式

模式 说明 适用场景
Single Tool 定义一个工具，Schema 包含所有字段 简单任务，输出结构固定
Multi-Tool 定义多个工具，让模型选择 复杂任务，需要不同输出类型
Chain 一个工具的输出作为下一个的输入 流水线处理
Router 先用路由工具分类，再调用对应工具 多分支逻辑

MCP (Model Context Protocol)

Anthropic 主导的 MCP 协议，使用 JSON Schema 定义 Tool 参数：

// MCP Tool 定义
{
  "name": "read_file",
  "description": "读取文件内容",
  "inputSchema": {
    "type": "object",
    "properties": {
      "path": {
        "type": "string",
        "description": "文件路径"
      }
    },
    "required": ["path"]
  }
}
"When you define a tool with a JSON Schema, you're not just documenting—you're constraining. And constraints are what make AI systems reliable."

— AI Engineering 实践

05实战案例

案例一：用户信息提取

需求：从非结构化文本中提取用户信息

输入

"我叫张三，今年28岁，邮箱是 zhangsan@email.com，喜欢编程和摄影"

Schema 设计
{
  "type": "object",
  "properties": {
    "name": { "type": "string", "description": "用户姓名" },
    "age": { "type": "integer", "minimum": 0, "maximum": 150 },
    "email": { "type": "string", "format": "email" },
    "interests": {
      "type": "array",
      "items": { "type": "string" }
    }
  },
  "required": ["name"]
}
输出
{
  "name": "张三",
  "age": 28,
  "email": "zhangsan@email.com",
  "interests": ["编程", "摄影"]
}

案例二：多意图分类

需求：分析用户查询，识别意图和实体

Schema 设计
{
  "type": "object",
  "properties": {
    "intents": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "type": {
            "enum": ["query", "command", "question"]
          },
          "action": { "type": "string" },
          "confidence": { "type": "number", "minimum": 0, "maximum": 1 }
        }
      }
    },
    "entities": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "type": { "enum": ["date", "person", "location", "product"] },
          "value": { "type": "string" },
          "normalized": { "type": "string" }
        }
      }
    }
  }
}

案例三：报告生成

需求：生成结构化的分析报告，便于后续处理

Schema 设计
{
  "type": "object",
  "properties": {
    "title": { "type": "string" },
    "summary": { "type": "string", "maxLength": 500 },
    "sections": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "heading": { "type": "string" },
          "content": { "type": "string" },
          "bulletPoints": {
            "type": "array",
            "items": { "type": "string" }
          }
        }
      }
    },
    "recommendations": {
      "type": "array",
      "items": { "type": "string" }
    }
  },
  "required": ["title", "summary"]
}

Schema 设计的关键决策

✗ 过于宽松

  • 不设置 additionalProperties: false
  • 所有字段都是可选的
  • 类型定义含糊（如 string 没有约束）
  • 数组不限制长度

✓ 刚好够用

  • 明确禁止额外字段
  • 核心字段设为 required
  • 字符串设置长度和格式
  • 数组设置合理的 min/max

06最佳实践与常见陷阱

设计原则

1. 描述即文档

每个字段都加 description，让 AI 理解字段含义

2. 类型优先

能枚举就不要自由文本，能用数字就不要字符串

3. 扁平优于嵌套

过深的嵌套增加 AI 出错概率，尽量控制在 3 层以内

4. 必填最小化

只把真正必须的字段设为 required，给 AI 灵活空间

常见陷阱

陷阱 问题 解决方案
忘记 additionalProperties AI 添加未预期字段 始终设置 false
嵌套太深 AI 容易搞错层级 控制在 3 层内
字段名歧义 AI 不确定用什么值 加 description 说明
枚举不完整 AI 被迫选择不合适的值 加 "other" 选项
缺少长度限制 AI 输出过长或过短 设置 minLength/maxLength

调试技巧

🔧 当 AI 输出不符合 Schema 时
  1. 简化 Schema：先用最小 Schema 测试，逐步添加约束
  2. 检查 description：确保字段描述清晰无歧义
  3. 查看原始输出：看看 AI 实际想输出什么
  4. 用验证工具：在线验证器快速定位问题
  5. 调整 prompt：在 prompt 中强调输出格式要求

Schema 版本管理

// 在 Schema 中添加版本信息
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://example.com/schemas/user-v2.json",
  "title": "User Profile",
  "description": "用户信息 Schema v2.0",
  ...
}

版本兼容性建议

  • 新增字段 → 小版本更新，向后兼容
  • 删除字段 → 大版本更新，breaking change
  • 修改类型 → 大版本更新
  • 收紧约束 → 大版本更新（可能 break 现有数据）
